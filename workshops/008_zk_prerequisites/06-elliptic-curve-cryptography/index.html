Elliptic Curve Equation

y² = x³ + ax + b

An elliptic curve is the set of all (x, y) points satisfying this equation, plus a special "point at infinity."

  • Different a, b values give different curves
  • BN128 uses: y² = x³ + 3   (a=0, b=3)
  • secp256k1 (Bitcoin): y² = x³ + 7   (a=0, b=7)
Set-theoretic view: The curve IS a set. Points are members if they satisfy the equation. The binary operator (point addition) is a function from this set to itself.
An elliptic curve is defined by this equation. For our purposes, a equals 0, so the equation simplifies to y squared equals x cubed plus b. The BN128 curve uses b equals 3. Remember from set theory: the curve is just a set of (x, y) pairs that satisfy the equation, plus the identity element (point at infinity). All our group theory applies directly.

Point Addition: Connect and Flip

To add points P and Q on the curve:

  1. Draw a line through P and Q
  2. The line intersects the curve at a third point R'
  3. Reflect R' over the x-axis to get R = P ⊕ Q
Key fact: A line that crosses an elliptic curve at two (non-tangent) points will ALWAYS cross at a third point, unless the line is perfectly vertical.

Vertical lines are the special case — they correspond to adding a point to its inverse, giving the point at infinity.

The geometric picture is elegant. Draw a line through two curve points, find the third intersection, then flip over the x-axis. This seems arbitrary, but the flip is necessary for the group laws to work. Without it, we'd get contradictions with associativity. The vertical line case gives us inverses: a point and its x-axis reflection always produce the point at infinity when added.

Point Doubling: P + P

What if both points are the same?

  • Take the tangent line at P (the derivative)
  • Find where it intersects the curve again
  • Reflect over x-axis → result is 2P
Point doubling enables fast scalar multiplication:
To compute 1000P, you only need ~10 doublings + ~10 additions
(binary decomposition: 1000 = 512 + 256 + 128 + 64 + 32 + 8) 
# 1000P = 512P + 256P + 128P + 64P + 32P + 8P
# 512P = double P nine times (P → 2P → 4P → ... → 512P)
# Total: ~10 doublings + ~6 additions ≈ 16 operations
# Much better than 999 additions!
Point doubling is what makes elliptic curve cryptography practical. To compute nP, we use the same trick as binary exponentiation: decompose n into powers of 2, compute each power of 2 by repeated doubling, then add them up. This gives us O(log n) operations instead of O(n). For a 256-bit scalar, that's about 256 doublings and 256 additions — perfectly feasible.

Point Addition Formula

Given P1 = (x1, y1) and P2 = (x2, y2):

λ = (y2 - y1) / (x2 - x1)
x3 = λ² - x1 - x2
y3 = λ(x1 - x3) - y1

For point doubling (P1 = P2):

λ = (3x1² + a) / (2y1)

Over finite fields, division becomes multiplication by the modular inverse. All operations stay exact!

Here are the actual formulas. Lambda is the slope of the line through the two points. For doubling, it's the derivative. Notice the formulas only use addition, subtraction, multiplication, and division — all of which work perfectly in finite fields. When we move to finite fields, division becomes multiplication by the modular inverse, and everything stays exact. No floating point, no rounding errors.

EC Points Form an Abelian Group

  1. Closed: adding two curve points gives another curve point (or point at infinity)
  2. Associative: (P ⊕ Q) ⊕ R = P ⊕ (Q ⊕ R) — proven algebraically
  3. Identity: the point at infinity O   (P ⊕ O = P)
  4. Inverse: inverse of (x, y) is (x, -y)   (reflection over x-axis)
  5. Commutative: P ⊕ Q = Q ⊕ P   (same line = same intersection)
Two points determine a unique line → one unique third intersection → same result regardless of order. Abelian!
We can verify all five properties! Closure comes from the geometry — lines always intersect the curve at a third point. The identity is the point at infinity — a mathematical construct that makes everything work. The inverse uses x-axis reflection: if the point (x, y) is on the curve, so is (x, -y), and their connecting line is vertical, giving the point at infinity. Commutativity is beautifully obvious: both points determine the same line, same third intersection, same result.

Elliptic Curves over Finite Fields

Instead of real numbers, use arithmetic mod p:

y² ≡ x³ + 3 (mod p) 
import libnum

def generate_points(mod):
    points = []
    for x in range(mod):
        y_sq = (x**3 + 3) % mod
        if libnum.has_sqrtmod_prime_power(y_sq, mod, 1):
            for y in libnum.sqrtmod_prime_power(y_sq, mod, 1):
                points.append((x, y))
    return points

# y² = x³ + 3 (mod 11) — a small example
points = generate_points(11)
print(f"Points on y² = x³ + 3 (mod 11):")
for p in sorted(points):
    print(f"  {p}")
print(f"Total: {len(points)} points + point at infinity = {len(points)+1}")
When we move to finite fields, the curve becomes a finite set of discrete points instead of a smooth curve. We iterate through all x values, compute x cubed plus 3 mod p, and check if the result has a modular square root. If it does, we have a valid point. For mod 11, we get a small set of points — small enough to enumerate but large enough to illustrate the concepts.

Point Addition over Finite Fields

def ec_add(x1, y1, x2, y2, a, p):
    """Add two points on y² = x³ + ax + b (mod p)"""
    if (x1, y1) == (None, None): return (x2, y2)
    if (x2, y2) == (None, None): return (x1, y1)
    
    if x1 == x2 and y1 == y2:
        # Point doubling
        lam = ((3 * x1**2 + a) * pow(2 * y1, -1, p)) % p
    elif x1 == x2:
        # Vertical line → point at infinity
        return (None, None)
    else:
        # Regular addition
        lam = ((y2 - y1) * pow(x2 - x1, -1, p)) % p
    
    x3 = (lam**2 - x1 - x2) % p
    y3 = (lam * (x1 - x3) - y1) % p
    return (x3, y3)

# Example: y² = x³ + 3 (mod 11)
print(ec_add(4, 10, 7, 7, 0, 11))  # Two points on the curve
Here's point addition implemented in Python for finite fields. Notice the only difference from real-number addition is using pow with -1 for modular inverse instead of regular division. The formulas are identical. This is the power of finite fields — the same algebra works, just with modular arithmetic. The None, None convention represents the point at infinity.

Generating the Cyclic Group

# y² = x³ + 3 (mod 11), generator G = (4, 10)
G = (4, 10)
p = 11

point = G
print(f"1G = {point}")
for i in range(2, 14):
    point = ec_add(point[0], point[1], G[0], G[1], 0, p)
    print(f"{i}G = {point}")

# Output:
# 1G  = (4, 10)     7G  = (0, 5)
# 2G  = (7, 7)      8G  = (1, 2)
# 3G  = (1, 9)      9G  = (7, 4)
# 4G  = (0, 6)      10G = (4, 1)
# 5G  = (8, 8)      11G = (None, None) ← identity!
# 6G  = (2, 0)      12G = (4, 10) ← back to G!
# The group has order 12 (11 points + point at infinity)
Starting from the generator point (4, 10), we can generate every point on the curve by repeated addition. The group has order 12 — note this is NOT the same as our modulus 11! After 11 additions we hit the point at infinity (the identity), and at 12 we cycle back to the generator. Every point has a unique "number" corresponding to how many times we added G. This numbering is the discrete logarithm.

Important: Order ≠ Modulus

Field modulus: the prime p we do arithmetic over (11 in our example)
Curve order: the number of points on the curve (12 in our example)
  • The modulus and order are different numbers
  • Adding the modulus to a scalar gives a different point
  • Adding the order to a scalar gives the same point (it wraps around)
  • For BN128: modulus ≈ 2254, order ≈ 2254 (close but different!)
The homomorphism works modulo the curve order, not the field modulus.
(a + b) mod order = aG + bG
This is a common source of confusion. The prime p that defines the field is NOT the same as the number of points on the curve. For our toy example, p is 11 but the order is 12. For BN128, both are roughly 2^254 but they're different numbers. The wrapping behavior — the homomorphism — works modulo the order, not the modulus. Getting this wrong would break everything.

The numbers involved are enormous — about 77 decimal digits. Ethereum has precompiled contracts for point addition, scalar multiplication, and pairing checks, making ZK verification gas-efficient.

Operations with py_ecc

from py_ecc.bn128 import G1, multiply, add, eq, neg

# Scalar multiplication: nG
p1 = multiply(G1, 5)    # 5G
p2 = multiply(G1, 10)   # 10G

# Point addition
p3 = add(p1, p2)        # 5G + 10G
p4 = multiply(G1, 15)   # 15G

# The homomorphism: 5G + 10G = 15G
assert eq(p3, p4)
print("5G + 10G = 15G ✓  — Homomorphism works!")

# Point negation (inverse)
neg_p1 = neg(p1)         # -(5G)
identity = add(p1, neg_p1)  # 5G + (-5G) = O
print(f"5G + (-5G) = {identity}")  # None (point at infinity)

# Associativity
a, b, c = multiply(G1, 3), multiply(G1, 7), multiply(G1, 11)
assert eq(add(add(a, b), c), add(a, add(b, c)))
print("Associativity verified ✓")
Here's the py_ecc library in action. Multiply does scalar multiplication, add does point addition, neg gives the inverse, and eq checks equality. The key line is the homomorphism check: 5G plus 10G equals 15G. This is the mathematical property that makes ZK proofs possible. We also verify associativity and the existence of inverses — all the group properties we studied.

The Discrete Logarithm Problem

Easy: Given x, compute P = xG   (scalar multiplication)
Hard: Given P, find x such that xG = P   (discrete logarithm) 
from py_ecc.bn128 import G1, multiply

# EASY: compute 12345G (instant)
secret = 12345
public_point = multiply(G1, secret)
print(f"12345 × G1 = {public_point}")

# HARD: given public_point, find 12345
# The only known approach for BN128 is essentially brute force
# (or baby-step giant-step, but still infeasible for 254-bit numbers)

# With ~2^254 possible values, checking each would take
# longer than the age of the universe, even at 10^18 ops/sec

This asymmetry is the foundation of elliptic curve cryptography and ZK proofs.

This is THE security assumption. Computing xG from x is trivial — a few hundred point doublings and additions. But recovering x from xG is computationally infeasible for large groups. The best known algorithm is roughly 2^128 operations for BN128, which is beyond the capability of any computer. This one-way property is what turns our homomorphism into a zero knowledge scheme.

Multiplication = Repeated Addition

There is no "point × point" operation on elliptic curves!
"Scalar multiplication" is just repeated point addition. 
from py_ecc.bn128 import G1, multiply, add, eq

# multiply(G1, 5) is really:
# G1 + G1 + G1 + G1 + G1

# Verify:
manual = G1
for _ in range(4):
    manual = add(manual, G1)

assert eq(manual, multiply(G1, 5))
print("5G = G+G+G+G+G ✓")

# Algebraic identity still works:
# (a+b)G = aG + bG  (because it's all addition)
a, b = 100, 200
assert eq(multiply(G1, a + b), add(multiply(G1, a), multiply(G1, b)))
print(f"({a}+{b})G = {a}G + {b}G ✓")
An extremely important clarification: when we say 5G, we really mean G plus G plus G plus G plus G. There is no multiplication of two curve points. Scalar multiplication is shorthand for repeated addition. This is why the notation can be confusing — it looks like multiplication but it's addition. The algebraic identity (a+b)G = aG + bG follows directly from associativity of the addition operation.

The Homomorphism in Practice

from py_ecc.bn128 import G1, multiply, add, eq, curve_order

# The core identity: (a + b) mod order ↔ aG + bG
# Works even with enormous numbers that overflow!

a = 2**300 + 21
b = 3**50 + 11

# Path 1: add integers, then map to curve
path1 = multiply(G1, (a + b))

# Path 2: map to curve, then add points
path2 = add(multiply(G1, a), multiply(G1, b))

assert eq(path1, path2)
print("φ(a + b) = φ(a) + φ(b) ✓")
print(f"Even with {(a+b).bit_length()}-bit numbers!")

# Also works with modular reduction
path3 = multiply(G1, (a + b) % curve_order)
assert eq(path1, path3)
print("Modular reduction gives same result ✓")
This is the homomorphism we've been building toward. With enormous numbers — hundreds of bits — the property still holds perfectly. Path 1 adds integers first, then maps to the curve. Path 2 maps each integer separately, then adds on the curve. Same result. The modular reduction doesn't even matter because the curve arithmetic handles overflow naturally. This is the mathematical magic that powers ZK proofs.

Point Inverses in Finite Fields

from py_ecc.bn128 import G1, multiply, neg, add, is_inf

field_mod = 21888242871839275222246405745257275088696311157297823662689037894645226208583

# The inverse has the same x but the y is the field inverse
for i in range(1, 4):
    point = multiply(G1, i)
    inv = neg(point)
    
    print(f"{i}G    = ({int(point[0])}, {int(point[1])})")
    print(f"-{i}G   = ({int(inv[0])}, {int(inv[1])})")
    
    # x values are the same
    assert int(point[0]) == int(inv[0])
    
    # y values are additive inverses: y + y' = field_mod
    assert int(point[1]) + int(inv[1]) == field_mod
    
    # Adding a point to its inverse gives the identity
    assert is_inf(add(point, inv))
    print(f"{i}G + (-{i}G) = point at infinity ✓\n")
Just like the 2D point group from Workshop 4, the inverse of an EC point has the same x coordinate but the y coordinate is negated — in finite field terms, the two y values sum to the field modulus. Adding a point to its inverse always gives the point at infinity, the identity element. The py_ecc library encodes the point at infinity as None.

Encoding Rational Numbers

from py_ecc.bn128 import G1, multiply, add, eq, curve_order

# In a finite field, 1/2 = modular inverse of 2
# This lets us encode "fractions" on the curve!

five_over_two = (5 * pow(2, -1, curve_order)) % curve_order
one_half = pow(2, -1, curve_order)

# 5/2 + 1/2 = 3 in regular math
result = add(multiply(G1, five_over_two), multiply(G1, one_half))
expected = multiply(G1, 3)

assert eq(result, expected)
print("(5/2)G + (1/2)G = 3G ✓")
print("Rational arithmetic works on the curve!")
We can encode rational numbers as field elements using modular inverses.
This extends the homomorphism to rational arithmetic.
We can even encode fractions. Five halves is encoded as 5 times the modular inverse of 2. Adding five-halves G and one-half G gives 3G — exactly as expected. This extends our homomorphism from integers to rationals, giving us even more computational power within the ZK framework. Division in the scalar space maps correctly to the elliptic curve space.

Building a ZK Proof

from py_ecc.bn128 import G1, multiply, add, eq

# ====== PROVER ======
# "I know x and y such that x + y = 15"
secret_x = 5
secret_y = 10

# Prover computes and publishes:
A = multiply(G1, secret_x)  # xG
B = multiply(G1, secret_y)  # yG
claim = 15

proof = (A, B, claim)

# ====== VERIFIER ======
# Verifier receives (A, B, 15) but does NOT know x or y
A, B, claimed_sum = proof

# Compute claimed_sum × G
expected = multiply(G1, claimed_sum)

# Check: A + B == claimed_sum × G?
if eq(add(A, B), expected):
    print("✓ VERIFIED: prover knows x, y with x + y = 15")
else:
    print("✗ REJECTED: proof is invalid")
Here's a working ZK proof! The prover has secret x equals 5 and y equals 10. They compute xG and yG and send those to the verifier along with the claim that x plus y equals 15. The verifier computes 15G and checks that xG plus yG equals 15G. By the homomorphism property, if this check passes, x plus y really does equal 15. The verifier is convinced without ever learning x or y individually.

ZK Proof: Linear Equations

from py_ecc.bn128 import G1, multiply, add, eq

# Prove: "I know x such that 23x = 161"
# (Solution: x = 7)

# ====== PROVER ======
secret_x = 7
commitment = multiply(G1, secret_x)  # xG

# ====== VERIFIER ======
# Verifier knows the equation 23x = 161
# They check: 23 × (xG) == 161G

lhs = multiply(commitment, 23)       # 23 × (xG) = 23xG
rhs = multiply(G1, 161)              # 161G

if eq(lhs, rhs):
    print("✓ VERIFIED: prover knows x such that 23x = 161")
    print("  (Verifier doesn't know x = 7!)")

# This works because:
# 23 × (xG) = 23xG = (23x)G = 161G ← by homomorphism
We can verify more complex relationships. Here the prover knows x equals 7 and proves 23 times x equals 161 without revealing x. The verifier multiplies the commitment xG by the scalar 23 and checks against 161G. Remember, multiplying a curve point by 23 is repeated addition — perfectly valid. This naturally extends to systems of linear equations, which is the basis for more sophisticated ZK proof systems.

Security Considerations

Our simple proofs are NOT fully zero knowledge!
If an attacker guesses x, they can verify by comparing xG to the commitment.
  • True ZK requires blinding factors (random masks)
  • Protocols like Groth16, PLONK add these safeguards
  • But the core mechanism — homomorphic verification — is what we showed
  • BN128 provides ~128 bits of security against discrete log
Real ZK proof systems add randomness and multi-round interactions to achieve full zero knowledge. The homomorphism we demonstrated is the foundation they build on.
Important caveat: our proofs are technically 'knowledge proofs' but not fully zero knowledge. If an attacker has a small number of candidates for x, they can check each one. Real ZK systems add random blinding factors that prevent this. But the core mathematical mechanism — the homomorphism — is exactly what we showed. Groth16 and PLONK build on this foundation with additional cryptographic machinery.

Treating EC as a Black Box

You don't need to understand the implementation of EC point addition.
You just need to know it follows group rules:
  • ✓ Adding EC points is closed — produces another EC point
  • ✓ Adding EC points is associative
  • ✓ There exists an identity (point at infinity)
  • ✓ Every point has an inverse
  • ✓ The group is cyclic — generated from G
  • ✓ Homomorphism: (a+b)G = aG + bG

This is like using a hash function — you don't need to know SHA-256's internals to use it correctly.

This is the most important takeaway. Just like you use hash functions without understanding their internals, you can use elliptic curves by understanding the interface — the group properties. Adding points gives a point. There's an identity. Each point has an inverse. The homomorphism works. That's all you need to build and understand ZK proof systems at a conceptual level.
