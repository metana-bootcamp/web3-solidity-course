Homomorphisms

Structure-preserving maps — the mathematical mechanism behind zero knowledge

Welcome to Workshop 5. Homomorphisms are where all our abstract algebra knowledge pays off. A homomorphism is a structure-preserving map between two groups. This is THE concept that makes zero knowledge proofs possible — proving you did a computation correctly without revealing the inputs.

What is a Homomorphism?

Given two groups (A, □) and (B, ■), a homomorphism from A to B is a function φ such that:

φ(ai □ aj) = φ(ai) ■ φ(aj)
for all ai, aj in A

In plain English:

  • Combine in A first, then map → same result as
  • Map each element first, then combine in B

The map "preserves" the group structure — operations in A correspond to operations in B.

This is the core definition. A homomorphism is a function between two groups that respects the operations. You can either operate first then map, or map first then operate — you get the same result either way. This is structure preservation. Think of it like a translation that preserves sentence structure between languages.

Homomorphism Visualized

   Group A (□)              Group B (■)
   ─────────               ─────────
   a_i ──────── φ ────────→ φ(a_i)
    □                          ■
   a_j ──────── φ ────────→ φ(a_j)
    ↓                          ↓
   a_i □ a_j                φ(a_i) ■ φ(a_j)
    │                          │
    └────── φ ────────→  φ(a_i □ a_j)
    
    Both paths give the SAME result!
Path 1: Combine in A, then apply φ
Path 2: Apply φ to each, then combine in B
Result: Same answer either way
The diagram shows two paths from the top to the bottom right. Path 1 goes down then right — combine first, then map. Path 2 goes right then down — map each element first, then combine. The fact that both paths reach the same destination is the definition of a homomorphism. This commutativity of the diagram is the whole point.

Example 1: Integers → Even Integers

Let φ(x) = 2x

  • A = (ℤ, +) all integers under addition
  • B = (2ℤ, +) even integers under addition
  • φ(x) = 2x
# Verify:
# φ(a + b) = φ(a) + φ(b)
# 2(a + b) = 2a + 2b  ✓

a, b = 3, 5
lhs = 2 * (a + b)  # = 16
rhs = 2*a + 2*b    # = 16
assert lhs == rhs

print(f"φ({a}+{b}) = φ({a+b}) = {lhs}")
print(f"φ({a})+φ({b}) = {2*a}+{2*b} = {rhs}")
# Both equal 16 ✓
The simplest homomorphism. Doubling is a homomorphism from all integers to even integers, both under addition. Doubling the sum equals the sum of doubled values. This is really just the distributive property, but it's a perfect first example of structure preservation.

Example 2: Strings → Integers

Let φ = length function (len)

A = (strings, concatenation) — a Monoid
B = (ℤ≥0, +) — a Monoid
φ(s) = len(s) 
# φ(concat(s1, s2)) = φ(s1) + φ(s2)
# len("Rare" + "Skills") = len("Rare") + len("Skills")

s1, s2 = "Rare", "Skills"
lhs = len(s1 + s2)           # len("RareSkills") = 10
rhs = len(s1) + len(s2)      # 4 + 6 = 10
assert lhs == rhs

print(f"len('{s1}' + '{s2}') = len('{s1+s2}') = {lhs}")
print(f"len('{s1}') + len('{s2}') = {len(s1)} + {len(s2)} = {rhs}")
# Both equal 10 ✓

The string length is a homomorphism from strings under concatenation to non-negative integers under addition.

This is a beautifully intuitive example. The length of the concatenation equals the sum of the lengths. You've been using this homomorphism your entire programming career without knowing it had a name. Note this works between Monoids, not just Groups — homomorphisms apply to any algebraic structure with a binary operator.

Example 3: Matrices → Integers (Determinant)

Let φ = det (determinant)

A = (2×2 integer matrices, ×) — a Monoid
B = (ℤ, ×) — a Monoid
φ(M) = det(M) 
import numpy as np

A = np.array([[1, 2], [3, 4]])
B = np.array([[5, 6], [7, 8]])
C = A @ B  # matrix multiplication

det_A = int(round(np.linalg.det(A)))  # -2
det_B = int(round(np.linalg.det(B)))  # -2
det_C = int(round(np.linalg.det(C)))  # 4

print(f"det(A×B) = {det_C}")
print(f"det(A) × det(B) = {det_A} × {det_B} = {det_A * det_B}")
assert det_C == det_A * det_B  # 4 = (-2)×(-2) ✓
The determinant is a homomorphism from matrices under multiplication to integers under multiplication. The determinant of a product equals the product of the determinants. This is a well-known linear algebra fact, but now you see it's really a homomorphism. Notice the binary operators are different — matrix multiplication on one side, integer multiplication on the other — but the structure is preserved.

The Exponentiation Homomorphism

A = (ℤ, +)  →  B = ({bi : i ∈ ℤ}, ×)

φ(x) = bx

φ(a + b) = ba+b = ba × bb = φ(a) × φ(b) 
base = 2

# φ(a + b) = φ(a) × φ(b)
a, b = 3, 5
lhs = base ** (a + b)        # 2^8 = 256
rhs = base**a * base**b      # 8 × 32 = 256
assert lhs == rhs

# Addition in integers maps to multiplication in powers!
print(f"2^({a}+{b}) = 2^{a+b} = {lhs}")
print(f"2^{a} × 2^{b} = {base**a} × {base**b} = {rhs}")
Addition on one side becomes multiplication on the other.
This is the basis of how elliptic curve scalar multiplication works!
This is arguably the most important homomorphism for understanding ZK proofs. Addition in the exponent corresponds to multiplication in the base. When we write aG plus bG equals (a+b)G on an elliptic curve, we're using exactly this pattern — just in additive notation instead of multiplicative. The shift from addition to multiplication through a homomorphism is the core mechanism of elliptic curve cryptography.

Important Clarifications

  • One-directional: φ maps A → B, but there's no requirement for B → A
  • Not always exists: arbitrary groups may have no homomorphism between them
  • Not necessarily bijective: φ doesn't need to hit every element of B
  • Trivial homomorphism: mapping everything to the identity always works (but is useless)
  • Multiple homomorphisms: there can be more than one φ between A and B
The trivial homomorphism: φ(a) = eB for all a.
It's valid (φ(a□b) = e = e ■ e = φ(a)■φ(b)) but tells us nothing useful.
A few important points about homomorphisms. They're one-directional — just because we can map A to B doesn't mean we can map B back to A. The trivial homomorphism that sends everything to the identity always exists but is useless. For ZK proofs, we need homomorphisms that are NOT trivial — ones that actually encode meaningful information about the computation.

Rationals → Finite Field

φ(a/b) = a × b-1 (mod p)

p = 17

def phi(num, den):
    return (num * pow(den, -1, p)) % p

# 1/3 + 3/5 = 14/15 in regular math
a = phi(1, 3)   # φ(1/3) 
b = phi(3, 5)   # φ(3/5)
c = phi(14, 15) # φ(14/15)

print(f"φ(1/3) = {a}")    # 6
print(f"φ(3/5) = {b}")    # 4
print(f"φ(14/15) = {c}")  # 10

# Check: φ(1/3) + φ(3/5) ≡ φ(14/15) (mod 17)?
print(f"φ(1/3) + φ(3/5) = {(a + b) % p}")  # 10
assert (a + b) % p == c  # ✓ Homomorphism preserved!
This one's from Workshop 2 but now we have the vocabulary to describe it properly. It's a homomorphism from rationals under addition to finite field elements under addition. The structure is preserved — adding rationals and then mapping gives the same result as mapping first and then adding in the field. This is actually used in ZK systems when encoding rational arithmetic.

Homomorphic Encryption

If φ is computationally difficult to invert,
then φ homomorphically encrypts the elements of A.
  • φ maps from "secret space" A to "public space" B
  • We can verify computations in B without knowing values in A
  • Because φ preserves structure, computations in B reflect computations in A
  • Because φ is hard to invert, nobody can recover the original values

This is the fundamental mechanism of zero knowledge proofs!

This is the big reveal. If phi is easy to compute but hard to invert, we get homomorphic encryption. The prover computes in the secret space A, maps results to public space B using phi, and the verifier checks the computation in B. The structure preservation guarantees correctness, and the difficulty of inverting phi guarantees privacy. This is EXACTLY what elliptic curves give us — the discrete logarithm problem makes phi hard to invert.

Zero Knowledge Proof: Addition

Claim: "I know two numbers x and y such that x + y = 15"

# Setup: φ maps integers to elliptic curve points
# φ(x) = x × G  (scalar multiplication by generator)
# This is hard to invert (discrete logarithm problem)

# Prover sends:
#   A = φ(x) = xG
#   B = φ(y) = yG
#   claim: x + y = 15

# Verifier checks:
#   A ■ B  ==?  φ(15)
#   xG + yG ==? 15G
#
# By the homomorphism property:
#   φ(x + y) = φ(x) + φ(y) = xG + yG
#
# If xG + yG = 15G, then x + y = 15 ✓

# The verifier is convinced WITHOUT knowing x or y!
Here's ZK in action. The prover has secret x and y that sum to 15. They compute xG and yG — mapping into elliptic curve space. The verifier checks that xG plus yG equals 15G. If it does, the verifier is convinced the sum is 15 without learning x or y individually. The discrete logarithm problem ensures the verifier can't extract x from xG. This is the essence of zero knowledge.

Zero Knowledge Proof: Scalar Multiplication

Claim: "I know a and b, and b = 5a"

# Prover sends: φ(a) and φ(b)
# i.e., aG and bG

# Verifier checks:
#   φ(a) + φ(a) + φ(a) + φ(a) + φ(a) ==? φ(b)
#   5 × (aG) ==? bG
#
# Remember: "5 × (aG)" is repeated ADDITION
# That's just 5aG

# If 5(aG) = bG, then 5a = b ✓

# Pseudocode:
# A = multiply(G1, a)     # prover computes aG
# B = multiply(G1, b)     # prover computes bG
# assert multiply(A, 5) == B  # verifier checks 5(aG) = bG
"Multiplication by a constant" is really repeated addition.
Repeated addition = scalar multiplication on the curve.
We can verify multiplicative relationships too. The prover claims b equals 5a. They send aG and bG. The verifier computes 5 times aG by repeated addition and checks if it equals bG. Note we're NOT multiplying two curve points — we're multiplying a curve point by a scalar, which is repeated addition. This distinction is crucial: there's no point-times-point operation on elliptic curves, only point-times-scalar.

Why Elliptic Curve Points?

What makes EC points perfect for φ?

  • Group structure: EC points form a finite cyclic abelian group
  • Homomorphism: φ(x) = xG preserves addition
  • Hard to invert: given xG, finding x is the discrete logarithm problem
  • Efficient: point addition and scalar multiplication are fast
  • Compact: points can be represented with ~256 bits
EC points give us a one-way homomorphism:
Easy: integer → EC point (compute xG)
Hard: EC point → integer (discrete log)
Elliptic curve points are the ideal choice for phi because they satisfy all the requirements simultaneously. The group structure ensures the homomorphism works. The discrete logarithm problem ensures it's hard to invert. And the operations are computationally efficient. This is why every major ZK proof system — Groth16, PLONK, STARKs — relies on elliptic curves or similar structures.

The Big Picture

"Elliptic curve points in a finite field under addition are a finite cyclic group, and integers under addition are homomorphic to this group."

You now understand every word of this sentence:

  1. EC points under addition: closed, associative, has identity, has inverses
  2. Finite cyclic: bounded number of elements, all generated from G
  3. Integers under addition: the familiar group we know
  4. Homomorphic: φ(a+b) = φ(a) + φ(b), i.e., (a+b)G = aG + bG

This one sentence encodes EVERYTHING you need to know about how ZK proofs work at a high level.

Take a moment to appreciate this. Four workshops ago, this sentence would have been completely opaque. Now you can parse every word. Elliptic curve points form a group — you know what that means. It's cyclic — generated from a single point. Integers map homomorphically to it — structure is preserved. And the one-way nature of this map is what gives us zero knowledge. This is the conceptual core of all ZK proof systems.

Demo: Homomorphism in Action

# Verify the homomorphism property in Python
# φ(a + b) = φ(a) + φ(b) where φ(x) = 2^x

base = 2

# Test with many pairs
import random
for _ in range(5):
    a = random.randint(1, 20)
    b = random.randint(1, 20)
    
    # Path 1: add then map
    path1 = base ** (a + b)
    
    # Path 2: map then multiply
    path2 = (base ** a) * (base ** b)
    
    assert path1 == path2
    print(f"a={a:2d}, b={b:2d}: "
          f"{base}^({a}+{b})={path1}, "
          f"{base}^{a}×{base}^{b}={path2} ✓")
Run this code to see the homomorphism property in action with many random pairs. Every pair satisfies the property. This is the same verification a ZK verifier performs, just with simple numbers instead of elliptic curve points. In Workshop 6, we'll do this with actual EC points using the py_ecc library.

Advanced: Rationals → Finite Field (Multiplicative)

p = 17

def phi(num, den):
    return (num * pow(den, -1, p)) % p

# Multiplicative homomorphism:
# φ(1/3 × 3/5) = φ(1/3) × φ(3/5)

# 1/3 × 3/5 = 1/5 in regular math
a = phi(1, 3)    # φ(1/3) = 6
b = phi(3, 5)    # φ(3/5) = 4  
c = phi(1, 5)    # φ(1/5) = 7

print(f"φ(1/3) × φ(3/5) = {a} × {b} = {(a*b) % p}")  # 24 mod 17 = 7
print(f"φ(1/5) = {c}")  # 7
assert (a * b) % p == c  # ✓ Works for multiplication too!
The same φ works for both additive and multiplicative homomorphisms.
This is because finite fields have both group structures.
The same mapping phi works as both an additive and multiplicative homomorphism. This is a consequence of finite fields having two compatible group structures. This is powerful — it means we can verify both addition and multiplication claims using the same encoding. However, verifying both at the same time is trickier and requires more sophisticated techniques like bilinear pairings.

Key Takeaways

  • Homomorphism: φ(a □ b) = φ(a) ■ φ(b) — structure-preserving map
  • One-directional: A → B, not necessarily B → A
  • Homomorphic encryption: if φ is hard to invert, it encrypts values
  • ZK proofs: prover maps secrets to EC space, verifier checks there
  • (a+b)G = aG + bG — the core identity of elliptic curve ZK
  • The discrete logarithm problem makes φ(x) = xG a one-way function
You now have the complete mathematical foundation for understanding ZK proofs!
These are the key ideas. Homomorphisms preserve structure. When the mapping is hard to invert, it becomes homomorphic encryption. ZK proofs exploit this by mapping secret computations to elliptic curve space where they can be verified without revealing the secrets. You now have all the abstract algebra needed to understand how ZK proofs work.
