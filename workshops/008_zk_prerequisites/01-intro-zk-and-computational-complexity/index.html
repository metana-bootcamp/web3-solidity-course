Part 1: What Are Zero Knowledge Proofs?

~10 minutes

The Core Idea

A Zero Knowledge Proof lets a prover convince a verifier that a statement is true —

without revealing any information about why it's true.

Analogy: Prove to a bouncer you're over 21 — without showing your ID, name, address, or exact age.
This is the elevator pitch. A ZKP is a cryptographic protocol between two parties. The prover knows a secret — some piece of information — and wants to convince the verifier that a claim about that secret is true. The verifier learns absolutely nothing beyond "yes, the claim is true." No data leaks. Think of it like proving you know a password without ever typing it.

Why ZKPs Matter in Web3

Use CaseWhat's ProvenWhat Stays Hidden
Privacy mixers"I deposited funds earlier"Which deposit is yours
zkRollups"All transactions are valid"Transaction details
Private voting"I'm eligible and this is my vote"Who you voted for
Identity"I meet the criteria"Personal data
Here are four real use cases. Privacy mixers let you withdraw funds without linking to your deposit. zkRollups batch thousands of transactions with a single proof — the chain doesn't need to re-execute them. Private voting and identity are emerging use cases. The common theme: prove a fact, hide the details.

Three Properties of ZKPs

  • Completeness — If the statement is true and both parties are honest, the verifier will be convinced
  • Soundness — If the statement is false, no cheating prover can trick the verifier (except with negligible probability)
  • Zero Knowledge — The verifier learns nothing beyond the truth of the statement
Discussion: Can you think of a situation in smart contracts where you'd want to prove something without revealing the underlying data?
These are the three formal properties. Completeness means the system works — honest provers succeed. Soundness means it's secure — liars get caught. Zero Knowledge means it's private — the verifier gains no extra information. All three must hold simultaneously. Ask students for examples — they might mention private auctions, salary proofs, credit checks on-chain.

Part 2: Computational Complexity

P, NP, PSPACE — ~15 minutes

To understand what ZK proofs can and cannot do, we need to classify problems by difficulty.

Now we shift to theory. This section is the most important of the workshop — it determines the boundaries of what zero knowledge proofs can achieve. The key question: how hard is it to solve a problem versus verify a solution?

Problems in P

Easy to solve AND easy to verify. Both in polynomial time.

Sorting a list

  • Solve: O(n log n) — mergesort
  • Verify: O(n) — check each element ≥ predecessor

Search in a list

  • Solve: O(n) — scan the list
  • Verify: O(1) — "7 is at index 4" → check index 4

Graph connectivity

  • Solve: O(n + e) — BFS
  • Verify: O(n) — follow the claimed path
Problems in P are the easy ones. We can find the answer efficiently AND check it efficiently. Sorting: mergesort runs in n-log-n, and verifying is just a linear scan. Searching: O(n) to find, O(1) to verify given the index. Graph connectivity: BFS to find a path, then just walk the path to verify. All polynomial time.

Problems in NP

Hard to solve but easy to verify.

Sudoku

  • Solve: Exponential search through combinations
  • Verify: O(n²) — check rows, columns, subgrids

3-coloring a map

  • Solve: Brute-force through exponential color assignments
  • Verify: O(n) — check each pair of neighbors
Key insight: All P problems are also NP problems. P ⊆ NP
NP is where things get interesting for ZK. These are problems where finding the answer might take exponential time — searching through millions of combinations — but once someone claims to have an answer, checking it is fast. Sudoku: exponentially many ways to fill the grid, but verifying a filled grid is trivial. 3-coloring: checking that no neighbors share a color is a simple loop. Note: every P problem is also NP — if you can solve it fast, you can certainly verify it fast.

Problems in PSPACE

Hard to solve AND hard to verify.

Optimal chess move

  • A computer says "move pawn to f3"
  • How do you verify that's optimal?
  • You can't — you'd need to replay the entire game tree

Regex equivalence

  • Are a+ and aa* equivalent?
  • For simple cases, yes — for arbitrary regexes, exponential to verify
PSPACE is the dead zone for ZK proofs. Even if someone gives you an answer, there's no short proof you can check. Chess: you'd have to explore the entire game tree to confirm a move is optimal. Regex equivalence: you'd have to test an exponential number of strings. There's no witness that makes verification fast.

The Hierarchy

PSPACE

Hard to solve, hard to verify

NP

Hard to solve, easy to verify

P

Easy to solve, easy to verify

Here's the big picture. P sits inside NP, which sits inside PSPACE. Each outer ring adds difficulty. ZK proofs live in the NP zone — they work for any problem where verification is efficient.

Summary Table

ClassTime to SolveTime to Verify
PPolynomialPolynomial
NPNo requirement (often exponential)Polynomial
PSPACENo requirementNo requirement
ZK Proofs only work for P and NP problems — problems where the solution can be verified efficiently. If you can't verify it efficiently, you can't create a ZK proof for it.
This is the single most important takeaway from this section. Memorize this: ZKPs require efficient verification. If verification is exponential, no ZK trick can help. This rules out things like proving you found the perfect chess strategy.

Part 3: Witnesses & Verification

~10 minutes

What is a Witness?

A witness is proof that you solved the problem correctly — the evidence that lets someone verify your solution.

ProblemWitness
Sorting a listThe sorted list
Finding a number in a listThe index where it appears
Path between graph nodesThe sequence of nodes
SudokuThe filled-in grid
3-coloringThe color assignment
A witness is just the information needed to verify a solution. For sorting, the witness IS the sorted list — hand it to someone and they can check it in O(n). For Sudoku, the witness is the completed grid. The key: a witness doesn't have to be the solution in its original form — it could be a solution to an alternative representation of the same problem.

The "Knowledge" in Zero Knowledge

The "knowledge" in ZKP refers to knowledge of the witness.

  • A ZK proof says: "I know a witness that satisfies the verification"
  • But it does NOT reveal what that witness is
1. Define the PROBLEM as a circuit
2. The prover has the WITNESS (solution)
3. The prover generates a PROOF that the witness satisfies the circuit
4. The verifier checks the PROOF — without seeing the witness
This is the pipeline. Step 1: express your problem as a circuit — we'll learn how in the next sections. Step 2: the prover plugs in their secret witness. Step 3: cryptographic magic generates a proof. Step 4: the verifier checks the proof. The verifier never sees the witness, yet they become convinced it's valid. The entirety of ZK research is about making steps 3 and 4 efficient, secure, and truly zero-knowledge.

ZKPs Can't Do Everything

  • ZKPs cannot help you find a solution — only prove you have one
  • ZKPs cannot work for PSPACE problems — no efficient verification mechanism
  • ZKPs can work for any P or NP problem — anything with efficient verification
Discussion: Consider a smart contract that verifies Merkle proofs. What is the witness? What is the statement being proven?
Important misconception to address: ZKPs don't solve hard problems, they prove solutions. If you found a Sudoku solution, ZK can prove you know it without showing it. But ZK can't find the solution for you. And for problems like optimal chess moves, where even verification is exponential, ZK is useless. Ask students the Merkle proof question: the witness is the leaf value plus the sibling hashes, and the statement is "this leaf is part of this tree."

Part 4: Boolean Circuits

~10 minutes

We need a universal language to express any verifiable problem.
Boolean circuits give us that.

Boolean Operators

OperatorSymbolDescription
AND∧True only when both inputs are true
OR∨True when at least one input is true
NOT¬Flips true ↔ false

Example formula with variables x1, x2, x3, x4:

out = (x1 ∨ ¬x2 ∨ ¬x3) ∧ (¬x2 ∨ x3 ∨ x4)

Quick refresher on Boolean logic. AND, OR, NOT — these are the only building blocks we need. Any problem in P or NP can be expressed using just these three operators. The formula at the bottom takes four Boolean variables and produces a true/false output. Finding values that make it true might be hard. Verifying a proposed assignment is easy — just plug in and evaluate.

Verifying a Boolean Formula

Given: out = (x1 ∨ ¬x2 ∨ ¬x3) ∧ (¬x2 ∨ x3 ∨ x4)

Proposed witness: x1=T, x2=F, x3=T, x4=F

Plug in: (T ∨ T ∨ F) ∧ (T ∨ T ∨ F)

= T ∧ T

= TRUE ✓

Finding the assignment → may be exponential
Verifying a proposed assignment → polynomial (just plug in!)
Here's verification in action. Someone gives us four values. We substitute them in, evaluate left to right, and get TRUE. That took constant time per clause. Finding those four values from scratch? For large formulas, that's an NP-hard search. But checking is always fast.

3-Coloring as a Boolean Circuit

For each territory, create Boolean variables for each color:

WA_G, WA_B, WA_R — (Western Australia: Green, Blue, Red)

Color constraint — exactly one color per territory:

(WA_G ∧ ¬WA_B ∧ ¬WA_R) ∨ (¬WA_G ∧ WA_B ∧ ¬WA_R) ∨ (¬WA_G ∧ ¬WA_B ∧ WA_R)

Boundary constraint — neighbors have different colors:

¬(WA_G ∧ SA_G) ∧ ¬(WA_B ∧ SA_B) ∧ ¬(WA_R ∧ SA_R)

Let's see a real example. To prove a valid 3-coloring of a map, each territory gets three Boolean variables — one per color. The color constraint says exactly one must be true. The boundary constraint says for each pair of neighbors, they can't both be the same color. AND all these constraints together, and you have a Boolean circuit. A satisfying assignment IS a valid 3-coloring. But notice: 3 variables per territory. That's verbose. Arithmetic circuits will fix this.

The Problem with Boolean Circuits

Boolean circuits are verbose for arithmetic operations.

  • Expressing a + b = c where a=8, b=4, c=12 requires:
  • Converting everything to binary (4 bits each = 12 variables)
  • Building a massive carry-chain formula
((a₄ ∧ b₄ ∧ c₄) ∨ (¬a₄ ∧ ¬b₄ ∧ c₄) ∨ ...) ∧
((a₃ ∧ b₃ ∧ ((a₂ ∧ b₂) ∨ (b₂ ∧ (a₁ ∧ b₁) ∨ ...
// ... continues for many more lines ...
Simple addition becomes a wall of Boolean logic.
This is why we prefer arithmetic circuits for ZK.
Here's the motivation for the next section. Try expressing "8 + 4 = 12" in Boolean. You need 4 bits per number, 12 Boolean variables, and a massive formula modeling binary addition with carries. It's technically correct but insanely verbose. Arithmetic circuits let us just write "a + b === c" directly.

Part 5: Arithmetic Circuits

~10 minutes

A simpler, more compact way to express problems for ZK proofs.

What is an Arithmetic Circuit?

A system of equations using only three operations:

  • Addition (+)
  • Multiplication (×)
  • Equality (===)

First example:

6 === x₁ + x₂
9 === x₁ * x₂

Satisfied by x₁ = 3, x₂ = 3

=== means assert equal, not assign.
Think: assertEq(left, right)
An arithmetic circuit is just a system of equations. Variables are called "signals." The triple-equals means "assert these are equal" — it's a constraint, not an assignment. The circuit is "satisfied" when ALL equations hold simultaneously. In our example, x1=3, x2=3 works because 3+3=6 and 3*3=9. The circuit `x === x + 1` has NO solution — it's unsatisfiable.

Constraining Values

How do we force a signal to be 0 or 1?

x(x - 1) === 0

Only x = 0 or x = 1 satisfies this!

How do we force a signal to be one of {1, 2, 3}?

0 === (1 - x) * (2 - x) * (3 - x)

A polynomial that has roots at exactly 1, 2, and 3.

This is a beautiful trick. x times (x minus 1) equals zero only when x is 0 or 1 — those are the roots of the polynomial. Extend the idea: (1-x)(2-x)(3-x) equals zero only when x is 1, 2, or 3. We're using the fundamental theorem of algebra to constrain signal values. This is used everywhere in ZK circuit design.

3-Coloring with Arithmetic Circuits

Instead of 3 Boolean variables per territory → 1 signal with values {1, 2, 3}

// Color constraints — each territory is colored 1, 2, or 3
0 === (1 - WA) * (2 - WA) * (3 - WA)
0 === (1 - SA) * (2 - SA) * (3 - SA)
0 === (1 - NT) * (2 - NT) * (3 - NT)
// ... one per territory

// Boundary constraints — neighbors have different colors
// If x ≠ y and both ∈ {1,2,3}, then x*y ∈ {2, 3, 6}
0 === (2 - WA*SA) * (3 - WA*SA) * (6 - WA*SA)
0 === (2 - WA*NT) * (3 - WA*NT) * (6 - WA*NT)
// ... one per border
Same 15 constraints as Boolean, but ⅓ the variables.
Look how clean this is compared to the Boolean version. Each territory is one signal: 1 for blue, 2 for red, 3 for green. The color constraint forces each signal into {1,2,3}. The boundary constraint is clever: if two neighbors have DIFFERENT colors from {1,2,3}, their product must be in {2,3,6}. If they're the SAME color, the product is in {1,4,9}. So we constrain the product to be 2, 3, or 6. Same logic, way fewer variables.

Boolean ↔ Arithmetic Translation

Every Boolean gate has an arithmetic equivalent
(when inputs are constrained to 0 or 1):

BooleanArithmetic
NOT u → tt === 1 - u
u AND v → tt === u * v
u OR v → tt === u + v - u*v
This is why arithmetic circuits are fully equivalent to Boolean circuits. NOT: flip 0 to 1 and 1 to 0 — that's subtraction from 1. AND: both must be 1, so multiplication works. OR: at least one is 1 — addition minus the overlap. Since we can translate any Boolean gate, and any NP problem can be a Boolean circuit, it follows that any NP problem can also be an arithmetic circuit.

Conversion Example

Boolean: out = (x ∧ ¬y) ∨ z

Step by step → Arithmetic:

// Constrain inputs to 0 or 1
x(x - 1) === 0
y(y - 1) === 0
z(z - 1) === 0

// ¬y       →  (1 - y)
// x ∧ ¬y   →  x * (1 - y)
// ... ∨ z  →  x*(1-y) + z - x*(1-y)*z

// Final:
out === x - x*y + z - x*z + x*y*z
Let's walk through a real conversion. Start with the Boolean formula: x AND (NOT y) OR z. First, constrain all inputs to binary. Then substitute: NOT y becomes 1 minus y. AND becomes multiplication. OR becomes the addition-minus-product formula. Simplify algebraically and you get that last line. Both circuits accept exactly the same inputs.

The ZK Pipeline

Real-world problem

↓

Arithmetic Circuit (system of constraints)

↓

Prover: "I know a witness that satisfies all constraints"

↓

ZK Proof: Verifier confirms without seeing the witness

All problems in P and NP can be expressed as arithmetic circuits.
This is the foundation everything else is built on.
Zoom out. This is the entire ZK pipeline in four lines. You take a real problem, express it as an arithmetic circuit, the prover provides the witness, and cryptographic machinery generates a proof the verifier can check. Everything in the next five workshops builds toward making this pipeline work: finite fields give us the number system, group theory gives us the algebra, and elliptic curves give us the cryptography.

Live Demo: Boolean vs Arithmetic

# Boolean circuit: out = (x AND NOT y) OR z
def boolean_circuit(x, y, z):
    return (x and not y) or z

# Arithmetic circuit equivalent (inputs constrained to 0 or 1)
def arithmetic_circuit(x, y, z):
    assert x * (x - 1) == 0, "x must be 0 or 1"
    assert y * (y - 1) == 0, "y must be 0 or 1"
    assert z * (z - 1) == 0, "z must be 0 or 1"
    return x - x*y + z - x*z + x*y*z

# Test all combinations
for x in [0, 1]:
    for y in [0, 1]:
        for z in [0, 1]:
            bool_result = 1 if boolean_circuit(x, y, z) else 0
            arith_result = arithmetic_circuit(x, y, z)
            assert bool_result == arith_result
            print(f"x={x}, y={y}, z={z} → {arith_result}")

print("All match — circuits are equivalent!")
If you have a few minutes, run this live. It exhaustively tests all 8 combinations of binary inputs and shows that the Boolean and arithmetic circuits produce identical results. This is the proof that the translation works. Students can try modifying the Boolean formula and re-deriving the arithmetic version as homework.

Key Takeaways

  1. ZKPs let you prove you know something without revealing it
  2. Only P and NP problems can have ZK proofs (solutions must be efficiently verifiable)
  3. Witnesses are the secret knowledge the prover has
  4. Arithmetic circuits are systems of equations (add, multiply, equals) that model any NP problem
  5. Arithmetic circuits are more compact than Boolean circuits for most ZK applications
Five things to walk away with. ZKPs prove knowledge without revealing it. They only work for problems with efficient verification. The witness is the secret. Arithmetic circuits are the language. And they're preferred over Boolean because they're more compact.
