What is a Set?

A well-defined collection of objects

Examples

  • {1, 2, 3, 4, 5}
  • {0, 1, 2, ..., p-1} ← a finite field!
  • All integers ℤ
  • All real numbers ℝ
  • All 3×3 matrices
  • All polynomials of degree ≤ 5

Rules

  • No duplicates: {a, a, b} = {a, b}
  • Order doesn't matter: {1,2,3} = {3,1,2}
  • Can be empty: { } = ∅
  • Membership is well-defined
You already know sets from programming — they're like arrays without duplicates or order. The key insight here is that ANYTHING can be a set: numbers, matrices, polynomials, even elliptic curve points. The rules we learn about sets apply to all of them uniformly.

Subsets & Number Systems

The classic number systems form a chain of subsets:

ℂ Complex Numbers
ℝ Real Numbers
ℚ Rational Numbers
ℤ Integers

ℤ ⊂ ℚ ⊂ ℝ ⊂ ℂ — Every integer is a rational number, but not every rational is an integer.

This is a classic example of subset relationships. All integers are rational numbers (you can write 5 as 5/1), all rationals are real, all reals are complex. The notation with the funny letters comes up constantly — Z for integers from the German word Zahlen, Q for quotient, R for real, C for complex. Get comfortable with these symbols.

Cardinality

The number of elements in a set

  • |{5, 9, 10}| = 3
  • |{0, 1, 2, ..., 10}| = 11
  • |𝔽₇| = 7 (finite field of order 7)
  • |ℤ| = ∞ (countably infinite)
  • |ℝ| = ∞ (uncountably infinite — a "bigger" infinity!)
In ZK proofs, we always work with finite sets, so cardinality = a specific number.
Cardinality is just a fancy word for size. The vertical bars are the notation. For us, the key point is that finite fields have finite cardinality — GF(p) has exactly p elements. The order of an elliptic curve group is also finite. We'll see "order" used interchangeably with "cardinality" for groups.

Ordered Pairs

Sets have no order, but we can create ordered pairs (tuples):

  • Set: {a, b} = {b, a}   (order doesn't matter)
  • Ordered pair: (a, b) ≠ (b, a)   (order matters!)
  • Two ordered pairs are equal iff both components match
Programmer translation: An ordered pair is a tuple.
(a, b) == (c, d) ⇔ a == c and b == d

Elliptic curve points are ordered pairs (x, y). The order matters!

You already know this from programming — a tuple where position matters. An elliptic curve point is literally an ordered pair of field elements. The point (3, 6) is different from (6, 3). This seems obvious but it's important to establish formally because Cartesian products are built from ordered pairs.

Cartesian Products

A × B = all possible ordered pairs (a, b) where a ∈ A, b ∈ B

A = {1, 2, 3}
B = {'x', 'y', 'z'}

# A × B
cartesian = [(a, b) for a in A for b in B]
# [(1,'x'), (1,'y'), (1,'z'),
#  (2,'x'), (2,'y'), (2,'z'),
#  (3,'x'), (3,'y'), (3,'z')]

print(f"|A × B| = {len(cartesian)}")  # 9 = |A| × |B|
A × Bxyz
1(1,x)(1,y)(1,z)

The Cartesian product is every possible pairing between two sets. If A has 3 elements and B has 3 elements, A cross B has 9 pairs. This is THE fundamental building block for defining functions and binary operators formally. When we say a function maps from A to B, we're really selecting specific pairs from A cross B.

Functions as Subsets of Cartesian Products

A function f: A → B is a subset of A × B where each element of A maps to exactly one element of B

✓ Valid function

# f(1) = y, f(2) = z, f(3) = x
f = {(1,'y'), (2,'z'), (3,'x')}
# Each input maps to exactly
# one output ✓

✗ Not a function

# 1 maps to BOTH p and q
bad = {(1,'p'), (1,'q'), (2,'q'),
       (3,'r')}
# Same input, different outputs ✗
Key insight: functions don't have to be "computable." A function is just a relation — a mapping — between sets. In ZK, we deal with functions between very exotic sets (like elliptic curve points).
This is the set-theoretic definition of a function: pick pairs from the Cartesian product such that each input has exactly one output. The crucial point is that functions are just mappings between sets. They don't have to be something you can write as a formula or a Python function. In ZK, we'll map between integers, polynomials, matrices, and elliptic curve points. They're all just sets with mappings between them.

Binary Operators

A binary operator takes two elements from a set and returns one element.

□ : A × A → A
  • Addition on integers: (3, 5) ↦ 8
  • Multiplication on rationals: (3.5, 2.0) ↦ 7.0
  • String concatenation: ("Rare", "Skills") ↦ "RareSkills"
  • Addition mod 7: (5, 4) ↦ 2
A closed binary operator always produces a result in the same set.
Division on integers is NOT closed: 1 ÷ 3 = 0.333... (not an integer!)
A binary operator is the core concept that everything builds on. It takes two elements and produces one. The key property is CLOSURE — the result must be in the same set. Addition is closed on integers because adding two integers gives an integer. But division is NOT closed on integers because dividing can give a fraction. Understanding closure is essential: when we say elliptic curve point addition is a closed binary operator, it means adding points always gives another point on the curve.

Constructing a Binary Operator

Addition mod 3 on the set {0, 1, 2}:

+012
0012
1120
2201
# Build the operation table
S = {0, 1, 2}
print("  | " + " ".join(str(b) for b in sorted(S)))
print("--+------")
for a in sorted(S):
    row = " ".join(str((a + b) % 3) for b in sorted(S))
    print(f"{a} | {row}")

# Verify closure: every entry is in {0, 1, 2} ✓
This table IS the binary operator, fully specified. Every cell contains an element from our original set, so it's closed. Notice the symmetry — this operation is commutative. Also notice zero is the identity element: the row and column for 0 just give back the other operand. This table approach works for any finite set and any binary operation.

The Algebraic Hierarchy

Magma — closed binary operator
Semigroup — + associative
Monoid — + identity element
Group — + every element has an inverse

Each level adds one more requirement. Groups are the star of the show for ZK.

This is the key hierarchy. Each level includes everything from the level above plus one more property. A Magma just needs closure. A Semigroup adds associativity. A Monoid adds an identity element. A Group adds inverses. ZK proofs primarily use Groups. When someone says 'this is a group,' you immediately know it has all four properties. This is powerful because you can reason about the structure without knowing the implementation of the binary operator.

Magma

A set with a closed binary operator. That's all!

# Example: positive integers under exponentiation
# a^b is always a positive integer ✓ (closed)
# But NOT associative:
a, b, c = 2, 3, 2
assert (a ** b) ** c != a ** (b ** c)
# (2³)² = 64 but 2^(3²) = 512

# And NOT commutative:
assert a ** b != b ** a
# 2³ = 8 but 3² = 9
A Magma is the least restrictive algebraic structure.
Only requirement: the operation stays in the set.
Positive integers under exponentiation form a Magma — the result is always a positive integer (closed), but it's neither associative nor commutative. This is a good example to remember because it shows that closure alone isn't very powerful. We need more structure to reason usefully about operations. That's why we keep adding requirements.

Semigroup

A Magma where the binary operator is associative

(a □ b) □ c = a □ (b □ c) 
# Example: non-empty strings under concatenation
# Closed: concat of two strings is a string ✓
# Associative: ("foo" + "bar") + "baz" == "foo" + ("bar" + "baz") ✓
a, b, c = "foo", "bar", "baz"
assert (a + b) + c == a + (b + c)  # "foobarbaz"

# But NOT commutative:
assert a + b != b + a  # "foobar" ≠ "barfoo"
# (That's OK — commutativity isn't required)

# No identity element (if no empty string allowed)
# No inverses: can't "un-concatenate"
String concatenation without the empty string is a perfect Semigroup example. It's closed and associative, but there's no identity element and no way to reverse a concatenation. Notice commutativity is NOT required at any level of the hierarchy. Some groups are commutative (called abelian), but it's not a requirement. Elliptic curve point addition happens to be commutative, which we'll see in Workshop 6.

Monoid

A Semigroup with an identity element

a □ e = e □ a = a   for all a in the set

Examples

Set + OpIdentity
ℤ under +0
ℤ under ×1
Strings + concat"" (empty)
Matrices + multiplyI (identity matrix)
# Positive integers under ×
# is a Monoid (identity = 1)
# but NOT a Group (no inverses!)

# 5 × 1 = 5  ✓ (identity)
# 5 × ? = 1  → ? = 1/5
# but 1/5 is not an integer!
# So no inverse exists
A Monoid adds the identity element. This is the element that does nothing — zero for addition, one for multiplication, the empty string for concatenation. Positive integers under multiplication form a Monoid: we have closure, associativity, and identity (1), but we DON'T have inverses because the inverse of 5 would be 1/5, which isn't a positive integer. We're just one property away from a Group.

Group — The Star of the Show

A Monoid where every element has an inverse

For every a, there exists a' such that a □ a' = e

Groups are where it all comes together. The domain matters enormously! Positive integers under addition aren't even a Monoid. Non-negative integers are a Monoid but not a Group. ALL integers under addition — that's a Group. And notice the last two rows: our finite field forms a Group under addition, and the non-zero elements form a Group under multiplication. This is exactly why finite fields are so useful for cryptography.

Abelian Groups

An abelian group is a group where the binary operator is commutative:
a □ b = b □ a for all a, b
  • Integers under addition: abelian (3 + 5 = 5 + 3)
  • Matrices under multiplication: NOT abelian (AB ≠ BA in general)
  • Finite field under addition: abelian
  • Elliptic curve points under point addition: abelian ← important!
"Abelian" is just a fancy word for commutative. Say abelian — you'll sound smarter.
All the groups we care about for ZK are abelian — meaning their binary operator is commutative. It's named after the mathematician Niels Henrick Abel. The fact that elliptic curve point addition is abelian is crucial because it means the order we add points doesn't matter. This lets us rearrange computations freely, which is essential for proof construction.

Why This Vocabulary Matters

Consider this statement:

"Elliptic curve points in a finite field under addition are a finite cyclic abelian group."

You now know this means:

  • ✓ Adding two points gives another point (closed)
  • ✓ Addition is associative
  • ✓ There's an identity element ("point at infinity")
  • ✓ Every point has an inverse
  • ✓ Addition is commutative (abelian)
  • ✓ The group has finite size
  • ✓ Every element can be generated from a single generator (cyclic)

You know 7 things about elliptic curves without knowing what they are!

THIS is the payoff. You don't know what elliptic curve point addition looks like, but you already know seven properties about it. This is like knowing a data structure is a balanced binary tree — you immediately know insertion is O(log n) without knowing how AVL rotations work. The vocabulary of abstract algebra lets us reason about ZK systems at a high level.

The Programmer's Analogy

# Abstract algebra is like interfaces/traits in programming!

class Group:
    """Any Group implementation must satisfy:
    - closed: combine(a, b) always returns an element in the group
    - associative: combine(combine(a,b), c) == combine(a, combine(b,c))
    - identity: combine(a, identity) == a
    - inverse: combine(a, inverse(a)) == identity
    """
    def combine(self, a, b): ...
    def identity(self): ...
    def inverse(self, a): ...

# Integers under + is a Group
# Finite fields under + is a Group
# EC points under point-add is a Group
# All use the SAME interface, different implementations!
Groups hide implementation details like interfaces hide implementations.
You can reason about behavior without knowing internals.
Think of abstract algebra as the math version of programming interfaces. Just like you can sort anything that implements Comparable, you can apply group theorems to anything that's a Group. The implementation of elliptic curve point addition is nightmarishly complex, but because it's a Group, we can reason about it using the same rules we apply to integer addition. This abstraction is what makes ZK proofs tractable.

Practice Problems

  1. Is the set {0, 1, 2, 3, 4} under multiplication mod 5 a Group? (Careful about 0!)
  2. Build the operation table for XOR on the set {0, 1}. What algebraic structure is it?
  3. Why can't strings under concatenation be a Group? Which property fails?
  4. Define the Cartesian product of {triangle, square} × {3, 4, 5}. Which subset defines the "number of sides" function?
  5. Is min(a, b) over all integers a Magma? A Semigroup? A Monoid? What about over positive integers?
  6. Consider 3-bit binary numbers under bitwise AND. Is this a Magma, Semigroup, Monoid, or Group?
These problems test each level of the hierarchy. Problem 1 is a classic trap — you need to exclude 0 for multiplication to form a group. Problem 3 is about inverses — there's no un-concat operation. Work through these to solidify the vocabulary before next workshop.
