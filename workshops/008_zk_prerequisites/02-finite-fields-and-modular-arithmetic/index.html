Why Finite Fields?

The Problem

  • Real number math → rounding errors
  • Floating point → imprecise
  • Infinite sets → can't enumerate
  • Cryptography needs exact answers

The Solution

  • Finite fields give exact arithmetic
  • Division produces integers, not fractions
  • Bounded set → practical for computers
  • Foundation of elliptic curve crypto
In regular math, dividing 1 by 3 gives 0.333... which we can't represent exactly. In a finite field modulo 7, dividing 1 by 3 gives exactly 5, because 5 times 3 mod 7 equals 1. This exactness is why all ZK computations happen in finite fields.

Modular Arithmetic Refresher

Think of a clock: after 12, it wraps back to 1

a mod n = remainder when a is divided by n

  • 17 mod 5 = 2   (17 = 3×5 + 2)
  • 23 mod 7 = 2   (23 = 3×7 + 2)
  • 100 mod 11 = 1   (100 = 9×11 + 1)

We say 17 is congruent to 2 modulo 5, written 17 ≡ 2 (mod 5)

Modular arithmetic is arithmetic with a wrapping point. Most of you have seen this before. The key idea is that we only care about the remainder. Two numbers are congruent if they have the same remainder when divided by the modulus.

Operations in Modular Arithmetic

All operations mod p (let p = 7):

OperationExampleResult
Addition5 + 4 mod 72
Subtraction2 - 5 mod 74
Multiplication3 × 4 mod 75
Exponentiation3² mod 72
Key rule: You can take mod at any intermediate step — the result is the same.
(a + b) mod p = ((a mod p) + (b mod p)) mod p
The operations work exactly as you'd expect. The important insight is that you can reduce at any step — this is what makes modular arithmetic efficient even with huge numbers. Notice subtraction can produce what looks like a negative number, but 2 minus 5 mod 7 is 4 because negative 3 mod 7 is 4.

What is a Finite Field?

A finite field 𝔽ₚ (or GF(p)) is:
The set {0, 1, 2, ..., p-1} with addition and multiplication mod p,
where p is prime.
  • Closed under +, −, ×, ÷ (except ÷ by 0)
  • Associative and commutative
  • Has additive identity (0) and multiplicative identity (1)
  • Every element has an additive inverse and a multiplicative inverse

Why prime? Because composite moduli lose the property that every non-zero element has a multiplicative inverse.

A field is a set where you can add, subtract, multiply, and divide (except by zero) and always stay in the set. The critical requirement is that p must be prime. If p were 6, then 2 times 3 equals 0 mod 6 — we'd have zero divisors, which breaks the field properties. With a prime modulus, every non-zero element has a unique multiplicative inverse.

Why Must p Be Prime?

# Composite modulus: p = 6
# 2 × 3 = 6 ≡ 0 (mod 6)  ← zero divisors!
# This means 2 has no multiplicative inverse mod 6

# Prime modulus: p = 7
# Every non-zero element has an inverse:
for a in range(1, 7):
    inv = pow(a, -1, 7)
    print(f"{a} × {inv} ≡ {(a * inv) % 7} (mod 7)")

# Output:
# 1 × 1 ≡ 1 (mod 7)
# 2 × 4 ≡ 1 (mod 7)
# 3 × 5 ≡ 1 (mod 7)
# 4 × 2 ≡ 1 (mod 7)
# 5 × 3 ≡ 1 (mod 7)
# 6 × 6 ≡ 1 (mod 7)
With a composite modulus like 6, 2 times 3 gives 0 even though neither is zero. This is a disaster for cryptography because it means we can't divide by 2 or 3. With prime 7, every single non-zero element pairs up with exactly one inverse. Notice the symmetry: the inverse of 2 is 4, and the inverse of 4 is 2.

Additive Inverses

The additive inverse of a is the element b such that a + b ≡ 0 (mod p)

-a ≡ p - a (mod p)
Element (mod 7)Additive InverseVerification
000 + 0 = 0
161 + 6 = 7 ≡ 0
252 + 5 = 7 ≡ 0
343 + 4 = 7 ≡ 0

Every element has a unique additive inverse. There are no "negative numbers" — just inverses.

In a finite field, there are no negative numbers. Instead, each element has an additive inverse that sums to zero. The inverse of a is simply p minus a. So in mod 7, "negative 3" is actually 4, because 3 plus 4 is 7, which is 0 mod 7. This is conceptually important — when we work with elliptic curves later, the idea of "negating" a point follows the same principle.

Multiplicative Inverses

The multiplicative inverse of a is the element b such that a × b ≡ 1 (mod p)

By Brute Force

# Find inverse of 3 mod 7
p = 7
for b in range(1, p):
    if (3 * b) % p == 1:
        print(f"3⁻¹ ≡ {b} (mod {p})")
# 3⁻¹ ≡ 5 (mod 7)

Python Built-in

# Python 3.8+ has built-in
# modular inverse
inv = pow(3, -1, 7)
print(inv)  # 5

# Verify
assert (3 * 5) % 7 == 1
The multiplicative inverse is the key operation that makes finite fields powerful. Python 3.8+ has this built in with the pow function's third argument. For 3 mod 7, the inverse is 5 because 3 times 5 is 15, and 15 mod 7 is 1. This is how we'll do "division" in finite fields.

Fermat's Little Theorem

If p is prime and a is not divisible by p, then:
ap-1 ≡ 1 (mod p)
  • Direct consequence: a-1 ≡ ap-2 (mod p)
  • This gives us a formula for computing inverses!
# Verify Fermat's Little Theorem
p = 7
for a in range(1, p):
    result = pow(a, p - 1, p)
    inv_fermat = pow(a, p - 2, p)
    inv_builtin = pow(a, -1, p)
    print(f"a={a}: a^(p-1)={result}, a^(p-2)={inv_fermat}, inv={inv_builtin}")
    assert result == 1
    assert inv_fermat == inv_builtin

This theorem is the theoretical foundation for efficient modular inverse computation.

Fermat's Little Theorem tells us that any non-zero element raised to the power p minus 1 gives 1. This is incredibly useful because it means we can compute inverses using exponentiation: the inverse of a is a to the power p minus 2. Under the hood, Python's pow function uses this approach combined with fast exponentiation. In ZK proofs, this appears everywhere when we need to compute division in circuits.

Division Without Precision Loss

In regular math: 1 ÷ 3 = 0.333... (imprecise!)

In 𝔽₇: 1 ÷ 3 = 1 × 3-1 = 1 × 5 = 5

Verify: 5 × 3 = 15 ≡ 1 (mod 7) ✓ 
p = 7

# "Divide" 1 by 3 in the finite field
result = (1 * pow(3, -1, p)) % p
print(f"1 ÷ 3 ≡ {result} (mod {p})")  # 5

# More examples
print(f"5 ÷ 2 ≡ {(5 * pow(2, -1, p)) % p} (mod {p})")  # 6
# Verify: 6 × 2 = 12 ≡ 5 (mod 7) ✓

print(f"10 ÷ 4 ≡ {(10 * pow(4, -1, p)) % p} (mod {p})")  # 6
# Verify: 6 × 4 = 24 ≡ 3 (mod 7)... wait, 10 mod 7 = 3 ✓
This is the magic of finite fields for ZK: division always produces an exact integer result. There are no fractions, no rounding, no precision loss. When we divide 1 by 3 in mod 7, we get 5, and we can verify this is correct because 5 times 3 mod 7 is 1. This property is essential for ZK proofs because circuits need exact computation.

Encoding Rational Numbers

Any rational number a/b can be mapped into 𝔽ₚ:

a/b ↦ a × b-1 (mod p) 
p = 17

# Map 1/3 into the field
one_third = (1 * pow(3, -1, p)) % p
print(f"1/3 ≡ {one_third} (mod {p})")  # 6

# Map 3/5 into the field
three_fifths = (3 * pow(5, -1, p)) % p
print(f"3/5 ≡ {three_fifths} (mod {p})")  # 4

# Verify: 1/3 + 3/5 = 14/15 in regular math
regular_sum = (1*5 + 3*3) / (3*5)  # 14/15
field_sum = (one_third + three_fifths) % p
fourteen_fifteenths = (14 * pow(15, -1, p)) % p

print(f"1/3 + 3/5 = {field_sum} (mod {p})")
print(f"14/15 = {fourteen_fifteenths} (mod {p})")
assert field_sum == fourteen_fifteenths  # ✓ Homomorphism!
This is a powerful result: rational number arithmetic over the reals is homomorphic to finite field arithmetic. The mapping preserves addition and multiplication. So 1/3 plus 3/5 equals 14/15 in regular math, and when we map each of those three fractions into the field, the relationship still holds. This is our first glimpse of a homomorphism, which we'll study in depth in Workshop 5.

Python galois Library

from galois import GF

# Create a finite field of order 7
GF7 = GF(7)

# Elements behave like regular numbers but are always mod 7
a = GF7(5)
b = GF7(3)

print(f"a + b = {a + b}")      # 1  (8 mod 7)
print(f"a * b = {a * b}")      # 1  (15 mod 7)
print(f"a / b = {a / b}")      # 4  (5 × 3⁻¹ mod 7)
print(f"a ** 2 = {a ** 2}")    # 4  (25 mod 7)
print(f"-a = {-a}")            # 2  (7 - 5)

# Works with larger primes too
GF101 = GF(101)
x = GF101(50)
print(f"50⁻¹ mod 101 = {GF101(1) / x}")  # multiplicative inverse

pip install galois — Useful for experimentation and verification

The galois library in Python wraps everything nicely. When you create a GF7 element, all operations automatically stay in the field. This is great for experimenting and verifying your understanding. Modular Square Roots

In regular math: √9 = ±3

In finite fields: √a mod p might have 0 or 2 solutions

p = 11

# Which elements have square roots mod 11?
for a in range(0, p):
    roots = [x for x in range(p) if (x * x) % p == a]
    if roots:
        print(f"√{a} mod {p} = {roots}")

# Output:
# √0 mod 11 = [0]
# √1 mod 11 = [1, 10]     ← 1² = 10² ≡ 1
# √3 mod 11 = [5, 6]      ← 5² = 6² ≡ 3
# √4 mod 11 = [2, 9]      ← 2² = 9² ≡ 4
# √5 mod 11 = [4, 7]      ← 4² = 7² ≡ 5
# √9 mod 11 = [3, 8]      ← 3² = 8² ≡ 9
Notice: the two roots always sum to p! (e.g., 5 + 6 = 11)
They are additive inverses of each other — just like ±3 in regular math.
Not every element in a finite field has a square root. Exactly half of the non-zero elements are quadratic residues, meaning they have square roots. When a square root exists, there are always exactly two solutions, and they're additive inverses of each other. This is directly analogous to how the square root of 9 is plus or minus 3 in regular math. We'll need modular square roots when working with elliptic curves in Workshop 6.

Testing for Square Roots: Euler's Criterion

a(p-1)/2 ≡ 1 (mod p) ⇒ a has a square root
a(p-1)/2 ≡ p-1 (mod p) ⇒ a does NOT have a square root 
p = 11

# Test which elements are quadratic residues
for a in range(1, p):
    euler = pow(a, (p - 1) // 2, p)
    has_sqrt = euler == 1
    print(f"a={a:2d}: a^((p-1)/2) = {euler:2d} → {'has' if has_sqrt else 'NO'} sqrt")

# Elements with square roots: 1, 3, 4, 5, 9  (5 out of 10)
# Elements without:           2, 6, 7, 8, 10  (5 out of 10)
# Exactly half!
Euler's criterion gives us a fast way to check if an element has a square root without actually computing it. This is important for performance: testing costs one exponentiation, while finding the root is more complex. Notice exactly half of non-zero elements are quadratic residues. This fact comes up in number theory proofs related to ZK schemes.

Polynomials in Finite Fields

Polynomials work the same way — just reduce coefficients mod p

from galois import GF, Poly

GF7 = GF(7)

# f(x) = 3x² + 2x + 5  over GF(7)
f = Poly([3, 2, 5], field=GF7)
print(f"f(x) = {f}")

# Evaluate at x = 4
print(f"f(4) = {f(GF7(4))}")  # 3(16) + 2(4) + 5 = 61 ≡ 5 mod 7

# g(x) = x + 3
g = Poly([1, 3], field=GF7)

# Polynomial addition
print(f"f + g = {f + g}")

# Polynomial multiplication
print(f"f * g = {f * g}")
Polynomials over finite fields are central to ZK proofs!
QAP, PLONK, and other ZK schemes encode computations as polynomial equations.
Everything we've learned about finite field arithmetic extends naturally to polynomials. The coefficients are finite field elements, and all operations reduce mod p. This is where it connects directly to ZK: modern proof systems like Groth16 and PLONK encode computations as polynomial equations over finite fields. The ability to evaluate, add, and multiply polynomials in finite fields is the heart of these systems.

Why Polynomials Matter for ZK

Schwartz-Zippel Lemma (Intuition)

If two polynomials of degree d agree at more than d points, they must be the same polynomial.
  • A degree-d polynomial has at most d roots
  • If p(x) - q(x) = 0 at d+1 points → p(x) = q(x)
  • Over a large field, random evaluation is almost certainly unique
ZK Insight: The prover encodes their computation as a polynomial.
The verifier checks it at a random point.
If it passes, the polynomial (and computation) is almost certainly correct.
This is a key insight: if I claim my polynomial equals yours, we don't need to compare all coefficients. We can just evaluate both at a random point. If they agree, the probability they're different is vanishingly small — at most d over p, where d is the degree. For a 256-bit prime, this probability is negligible. This is the core verification trick in most ZK proof systems.

Linear Systems in Finite Fields

Solve 3x + 5y ≡ 2 (mod 7) and x + 4y ≡ 6 (mod 7)

p = 7

# 3x + 5y ≡ 2 (mod 7)
# x + 4y ≡ 6 (mod 7)

# From equation 2: x ≡ 6 - 4y (mod 7)
# Substitute into equation 1:
# 3(6 - 4y) + 5y ≡ 2 (mod 7)
# 18 - 12y + 5y ≡ 2 (mod 7)
# 4 - 7y ≡ 2 (mod 7)    (18 mod 7 = 4)
# 4 ≡ 2 (mod 7)         (-7y mod 7 = 0)
# Hmm, let's try different coefficients...

# Actually: 2x + 3y ≡ 1 (mod 7), 5x + y ≡ 4 (mod 7)
# From eq 2: y ≡ 4 - 5x (mod 7)
# Sub: 2x + 3(4 - 5x) ≡ 1 → 2x + 12 - 15x ≡ 1 → -13x ≡ -11
# → x ≡ 11 × pow(13, -1, 7) ≡ 4 × pow(6, -1, 7) ≡ 4 × 6 ≡ 3 (mod 7)
# y ≡ 4 - 5(3) ≡ 4 - 15 ≡ -11 ≡ 3 (mod 7)
# Verify: 2(3) + 3(3) = 15 ≡ 1 ✓, 5(3) + 3 = 18 ≡ 4 ✓
print("x = 3, y = 3")
Linear algebra works identically in finite fields — you just need to use modular inverses instead of regular division. Gaussian elimination works fine; you just replace division with multiplication by the modular inverse. This extends to matrices as well. ZK circuits often reduce to systems of equations over finite fields.

Live Demo: Complete Field Operations

p = 23  # A prime

print("=== Finite Field GF(23) ===")
print(f"\nAdditive inverses:")
for a in [1, 5, 11, 22]:
    print(f"  -{a} ≡ {(-a) % p} (mod {p})")

print(f"\nMultiplicative inverses:")
for a in [2, 3, 7, 11]:
    inv = pow(a, -1, p)
    print(f"  {a}⁻¹ ≡ {inv} (mod {p})  [verify: {a}×{inv}={a*inv}≡{(a*inv)%p}]")

print(f"\nDivision:")
for (a, b) in [(10, 3), (1, 7), (15, 4)]:
    result = (a * pow(b, -1, p)) % p
    print(f"  {a} ÷ {b} ≡ {result} (mod {p})  [verify: {result}×{b}={(result*b)%p}≡{a%p}]")

print(f"\nFermat's Little Theorem:")
for a in [2, 5, 17]:
    print(f"  {a}^{p-1} ≡ {pow(a, p-1, p)} (mod {p})")  # Always 1
Let's walk through this demo together. This shows all the field operations we've discussed. Run this code yourself and modify the prime to see how the results change. Try p equals 7, then 11, then 101. The same relationships hold regardless of the field size.

Real-World Field Sizes

In practice, ZK proofs use enormous primes:

# The BN128 curve (used by Ethereum) has field modulus:
p = 21888242871839275222246405745257275088696311157297823662689037894645226208583

# That's a 254-bit number!
print(f"Bits: {p.bit_length()}")  # 254

# But arithmetic still works identically:
a = 12345678901234567890
b = 98765432109876543210

print(f"a + b mod p = {(a + b) % p}")
print(f"a × b mod p = {(a * b) % p}")
print(f"a / b mod p = {(a * pow(b, -1, p)) % p}")
print(f"a^(p-1) mod p = {pow(a, p - 1, p)}")  # Still 1!
Same rules, same operations — just much bigger numbers.
The Ethereum BN128 curve works in a 254-bit prime field. That's a number with about 77 decimal digits. But everything we learned today applies identically — addition, multiplication, inverses, Fermat's theorem. The only difference is performance, which is why optimized libraries exist. Python handles arbitrary precision integers natively, so you can experiment with these large fields directly.

Key Takeaways

  • Finite field 𝔽ₚ = {0, 1, ..., p-1} with mod-p arithmetic (p prime)
  • Every non-zero element has a multiplicative inverse
  • Division is exact — no fractions, no rounding
  • Fermat's Little Theorem: ap-1 ≡ 1, so a-1 = ap-2
  • Polynomials work over finite fields and are the basis of ZK
  • Modular square roots exist for exactly half of non-zero elements
Everything in ZK proofs happens inside finite fields. Master this, and the rest follows.
These are the core concepts to remember. Finite fields give us exact arithmetic that's computationally bounded —perfect for cryptography. Every ZK proof system operates over a finite field. Understanding how arithmetic works in these fields is the single most important prerequisite for understanding ZK proofs.
